# SGX UI - Test Coverage Plan

## Overview

This document outlines the unified test coverage setup for the entire SGX monorepo project. The coverage configuration has been optimized to aggregate test results from all packages and apps into a single coverage folder at the project root, enabling streamlined SonarQube integration.

---

## Current Test Coverage Setup

### Architecture

```
sgx-ui (root)
├── coverage/                          ← Unified coverage output folder (SonarQube scans here)
├── vitest.workspace.ts                ← Root workspace configuration
├── packages/
│   ├── shared/vitest.config.ts
│   ├── api-client/vitest.config.ts
│   ├── query-client/vitest.config.ts
│   └── ui-components/vitest.config.ts (2 existing tests)
└── apps/
    ├── auth-app/vitest.config.ts
    ├── host-app/vitest.config.ts
    └── index-studio-app/vitest.config.ts
```

### Key Features

- **Unified Coverage Output**: All packages output coverage to `./coverage` at root level
- **Single Report for SonarQube**: One HTML report and one LCOV report for entire project
- **V8 Coverage Provider**: Fast, modern coverage collection
- **Standard Reporters**: LCOV, HTML, and Text formats
- **Global Test Setup**: Shared test utilities via `config/vitest/setup.ts`

## Configuration Details

### Coverage Provider & Reporters

| Reporter | Purpose                                  | Location                |
| -------- | ---------------------------------------- | ----------------------- |
| **LCOV** | SonarQube integration (machine-readable) | `./coverage/lcov.info`  |
| **HTML** | Interactive coverage dashboard           | `./coverage/index.html` |
| **Text** | Console output summary                   | Console during test run |

### Configured Coverage Exclusions

Each package excludes from coverage reporting:

- Entry point files (`index.ts`, `main.tsx`)
- Type definitions (`*.d.ts`)
- Test files themselves (`*.test.ts`, `*.spec.ts`)
- Style files (`styles/**`)
- Declarations and environment files

This ensures only source code is measured for coverage.

## Usage Guide

### Running Tests

```bash
# Run all tests across all packages
npm test

# Watch mode - re-run tests on file changes
npm run test:watch

# Run all tests with coverage report
npm run test:coverage

# Watch mode with coverage
npm run test:coverage:watch

# View HTML coverage report
npm run coverage:report
```

### Adding Tests to a Package

1. Create test files in `src/**/*.test.ts` or `src/**/*.spec.ts`
2. Run `npm run test:coverage`
3. Coverage automatically aggregates to root `./coverage` folder
4. No additional configuration needed

### Example Test File

```typescript
import { describe, it, expect } from "vitest";
import { render, screen } from "@testing-library/react";
import MyComponent from "./MyComponent";

describe("MyComponent", () => {
  it("renders correctly", () => {
    render(<MyComponent />);
    expect(screen.getByText(/test/i)).toBeInTheDocument();
  });
});
```

## SonarQube Integration

### Pre-requisite

The `./coverage/lcov.info` file generated by `npm run test:coverage` is the key SonarQube artifact.

### SonarQube Configuration (sonar-project.properties)

```properties
sonar.projectKey=sgx-ui
sonar.projectName=SGX UI
sonar.sources=./apps,./packages
sonar.tests=./apps,./packages
sonar.test.inclusions=**/*.test.ts,**/*.test.tsx,**/*.spec.ts,**/*.spec.tsx
sonar.coverage.exclusions=**/node_modules/**,**/dist/**,**/coverage/**
sonar.lcov.reportPaths=./coverage/lcov.info
```

### Workflow

1. Run: `npm run test:coverage`
2. Commit to repository
3. SonarQube scans `./coverage/lcov.info` for quality gates
4. Single unified coverage metric for entire project

## Troubleshooting

### Issue: Coverage folder not created

**Solution**: Ensure `npm run test:coverage` is run. Regular `npm test` doesn't generate coverage.

### Issue: Specific package not included

**Solution**: Check that `vitest.config.ts` exists in the package and is listed in `vitest.workspace.ts`.

### Issue: Coverage reports show 0%

**Possible causes**:

- No test files in package (this is normal for initialization)
- Test files not matching `**/*.{test,spec}.{ts,tsx}` pattern
- Source code in excluded paths

### Issue: Different coverage per package vs. aggregate

**Explanation**: Each package reports only its own coverage. The aggregate shows overall project coverage across all measured packages.

## Best Practices

1. **Aim for >80% coverage** on critical paths
2. **Exclude only what's necessary** (entry points, styles, types)
3. **Run coverage before commits** to catch regressions
4. **Review HTML reports** at `./coverage/index.html` for gaps
5. **Keep tests fast** - use mocking for external dependencies
6. **Use `.test.ts` suffix** consistently for all test files
